事件背景

當生產力工具變成攻擊媒介

許多人把 Obsidian 當成私密的第二大腦，記錄工作筆記、投資研究、加密貨幣持倉。 然而正是這份「私密感」，讓它成為攻擊者精心選中的誘餌平台。

資安公司 Elastic Security Labs 近期發布報告，揭露一個針對性極強的攻擊行動，代號 REF6598。 攻擊者假冒創投公司，鎖定加密貨幣與金融從業者，透過精心設計的社交工程手段， 讓受害者主動安裝含有惡意設定的 Obsidian vault，最終在受害者裝置上部署完整功能的遠端存取木馬（RAT）—— 而這一切都透過 Obsidian 的合法外掛機制完成。

Obsidian CEO kepano 的官方回應強調，這是一起社交工程攻擊，需要受害者「主動略過多個安全警告」才能成功， 且目前尚無實際用戶受害報告。但他也承認，安全性重大更新即將推出。

— kepano，Obsidian CEO，HN 討論串

攻擊手法解析

五步完成滲透：從 LinkedIn 到完整系統控制

這不是廣撒網的大規模攻擊，而是高度針對性的定向獵殺。攻擊流程環環相扣， 每一步都仰賴受害者的信任與配合：

1
LinkedIn 釣魚接觸

攻擊者建立虛假創投公司帳號，主動聯繫加密貨幣交易員、DeFi 開發者、金融分析師等高價值目標。 初始接觸通常以商業合作或投資機會為由。

2
移至 Telegram 建立信任

對話轉移到 Telegram 後，多個假冒「合夥人」帳號會輪番出現， 扮演不同角色（法務、技術顧問、另一位投資人），透過群組互動強化可信度。

3
要求使用 Obsidian 作為「協作工具」

攻擊者要求受害者使用 Obsidian 管理「共享資料庫」，並提供一組「雲端 vault」帳號。 這個要求看似合理——許多真實的加密貨幣基金確實用 Obsidian 做內部文件管理。

4
啟用社群外掛同步，惡意設定入侵

受害者啟用社群外掛同步後，預先植入惡意設定的 Shell Commands 外掛自動同步到本地。 這個外掛本身是合法的——它存在於 Obsidian 官方外掛市場，功能是讓使用者自動執行 shell 指令。 再配合 Hider 外掛隱藏相關 UI 元素，受害者幾乎不會發現異常。

5
下次開啟 vault 時自動觸發

惡意指令設定為「vault 開啟」事件觸發，受害者下次打開 Obsidian 的瞬間， 攻擊鏈自動啟動，完全不需要任何額外互動。

技術深潛

PHANTOMPULSE：一隻有品味的 RAT

Elastic 將這個攻擊鏈中使用的惡意軟體命名為 PHANTOMPULL（載入器）和 PHANTOMPULSE（完整 RAT）。以下是完整的攻擊鏈分解：

階段 名稱 描述
Stage 1 Shell Commands 設定 Base64 編碼的 PowerShell（Windows）或 AppleScript（macOS），由外掛設定觸發執行
Stage 2 下載腳本 連線至 C2 伺服器 195.3.222[.]251 下載後續 Payload
Stage 3 PHANTOMPULL 64-bit Windows 載入器（syncobs.exe），負責部署與執行最終 RAT
Stage 4 PHANTOMPULSE 完整功能的 Windows RAT，具備遠端控制、資料竊取、持久化能力

區塊鏈 C2 解析：攻守兼具的設計

PHANTOMPULSE 其中一個最有趣的技術特徵，是利用 Ethereum Blockscout 讀取鏈上交易資料來解析 C2 位址。 攻擊者將 C2 伺服器的 IP 或域名編碼為區塊鏈交易的附加資料，惡意軟體透過查詢公開區塊鏈 API 取得最新的 C2 位址。

這個設計的優點在於：傳統的 C2 封鎖手段（黑名單 IP、封鎖域名）對它幾乎無效， 因為它隱藏在不可更改的公開帳本後面。然而，這同時也留下了一個有趣的防禦機會—— 防守方可以用相同的機制監控鏈上資料，甚至嘗試劫持 C2 解析流程。

疑似 AI 輔助撰寫的跡象

Elastic 研究員注意到 PHANTOMPULSE 的 debug 字串異常詳細， 幾乎每個函式都有完整的操作說明字串，這種風格與 AI 輔助生成的程式碼高度吻合。 這也印證了一個趨勢：惡意軟體開發的技術門檻正在被 AI 工具大幅降低。

其他反分析技術

逃避偵測的手段

Runtime API Hashing：在執行時動態解析 Windows API 名稱，讓靜態分析難以判斷程式呼叫了哪些系統函式。

偽造完整性檢查：模仿合法程式的行為模式，對抗行為分析引擎。

定時器回呼迴避沙盒：利用 Windows 定時器機制偵測沙盒環境， 在檢測到自動化分析環境時暫停惡意行為。

macOS 版本

攻擊者同時準備了 macOS 版本，流程類似但使用 AppleScript（osascript）作為初始觸發器， 並建立 LaunchAgent 達成持久化。備援 C2 解析機制則更簡單粗暴—— 先嘗試硬編碼的域名，若失敗，直接抓取公開 Telegram 頻道內容作為死置解析器（dead-drop resolver）。

精美的 C2 管理面板

這個攻擊行動還附帶了一個以 Cloudflare 保護的「Phantom Panel」C2 Web 管理介面。 攻擊者顯然不是業餘玩家——有完整的基礎設施、跨平台 Payload、外加一個像樣的管理後台。

社群反應

HN 社群的四種聲音

這個事件在 Hacker News 引發了大量討論，社群大致分裂成四個陣營：

早就說過了

「我當年就說過了」派

用戶 ibash 指出，幾年前他就在 Discord 警告過 Obsidian 外掛有完整磁碟存取權限， 當時不僅沒人理，還被社群成員噴。「現在說太晚了。」 fooqux 補充：「有開放性」跟「外掛可以存取整顆磁碟」是兩回事，不能混為一談。

辯護派

「開放性是特色」派

PurpleRamen 等人認為開放性正是 Obsidian 的核心優勢， 不能因為一個社交工程攻擊就把整個外掛生態關掉。 真正應該強化的是社交工程防護，以及讓安全警告更難被忽略。

架構批評

「Electron 原罪」派

多人指出根本問題：Obsidian 是 Electron App，外掛跑在 Node.js 環境， 天生就有完整 OS 存取權限。有人拿 VS Code 延伸套件類比—— 沙盒化是可能的解法，但會破壞太多現有外掛的功能。

警告失效

「UI 警告設計太差」派

雖然這個攻擊確實需要受害者略過多個安全警告， 但有聲音指出：Obsidian 現有的安全警告設計得太容易讓人點過去， 幾乎沒有阻擋作用。用恐嚇語氣反而讓使用者習慣性地無視。

深層問題

Obsidian 的結構性困境

這個事件暴露的不只是一個「壞外掛」的問題，而是 Obsidian 外掛架構的根本性風險。 理解這個問題，需要先了解 Obsidian 外掛的本質：

安裝 Obsidian 社群外掛，在技術上等同於執行任意 npm 套件。 外掛運行在 Node.js 環境，擁有完整的作業系統存取權限—— 可以讀寫任意檔案、發起網路請求、執行系統指令。沒有沙盒，沒有權限隔離。

這和 VS Code 延伸套件的問題幾乎一模一樣。VS Code 的延伸套件同樣可以做到任何 Node.js 程式能做的事。差別在於，VS Code 的使用者大多是開發者， 對「安裝套件就是執行任意程式碼」這個概念有更直覺的認知； 而 Obsidian 的使用者則包含大量非技術用戶，他們把外掛當成「App Store 的 App」來理解， 以為有官方審核就代表安全。

這個認知落差，才是真正的危險所在。攻擊者甚至不需要上架惡意外掛—— 他們利用的是 合法外掛的合法功能，只是把惡意指令藏進設定檔裡。 這讓傳統的外掛審核機制形同虛設。

# Obsidian Shell Commands 外掛設定（簡化示意） # 惡意版本：vault 開啟時自動執行 Base64 編碼的 PowerShell { "commands": { "vault-open-trigger": { "shell_command": "powershell -enc [BASE64_ENCODED_PAYLOAD]", "events": ["on-obsidian-starts"], "enabled": true } } }

這段設定不包含任何惡意程式碼本身，只是告訴 Shell Commands 外掛 「在程式啟動時執行這個 PowerShell 指令」。惡意 Payload 在 Base64 解碼後才出現， 靜態分析幾乎不可能直接發現問題。

自保指南

如何保護自己

好消息是：這個攻擊需要大量的社交工程配合，不是隨機的大規模攻擊。 理解攻擊流程後，防護其實不難。

  • ✓
    絕對不要使用陌生人提供的 vault 或帳號 任何人提供你「現成的 Obsidian vault」或「雲端同步帳號」， 都應立即提高警覺。這是這個攻擊鏈的核心前提。
  • ✓
    啟用社群外掛前仔細審查設定 若必須啟用 Shell Commands 等可執行外掛，開啟後立即檢查設定， 確認沒有任何不認識的「vault 事件」觸發指令。
  • ✓
    對 LinkedIn/Telegram 商業接觸保持高度警戒 主動聯繫你提供「合作機會」、並要求你安裝特定工具的陌生人， 永遠應先驗證身份。搜尋公司名稱、查核域名、撥打公開電話。
  • ✓
    定期稽核已安裝的外掛清單 移除所有不再使用的外掛，並定期到外掛資料夾 （.obsidian/plugins/） 確認沒有出現陌生的目錄。
  • ✓
    使用端點安全工具監控異常網路連線 Shell Commands 外掛觸發的 PowerShell 連外行為， 在有完善端點保護的環境中可以被偵測到。企業用戶應確保端點保護覆蓋 Obsidian 的執行路徑。

結論

信任模型的崩壞

REF6598 事件真正令人警醒的，不是技術手法有多高明—— 攻擊者使用的全部都是合法工具、合法外掛、合法的 API。 真正的武器是使用者的信任：信任朋友推薦的工具、信任「只要點個 OK 就好」的警告框、 信任一個看起來專業的 Telegram 群組。


在工具越來越開放、整合越來越緊密的今天， 安全邊界的責任不能只靠軟體商的審核機制。 每一個使用者，尤其是掌管高價值資產的金融與加密貨幣從業者， 都需要對「安裝任何東西」這個動作，建立起與安裝系統程式同等的警覺心。
